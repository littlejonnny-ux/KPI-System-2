## 1. Назначение системы

Система предназначена для управления индивидуальными KPI-картами сотрудников уровней CEO-1 и CEO-2 микрофинансовой компании с активами 10 млрд руб.

Ключевые цели:
- Формализовать и автоматизировать систему KPI для топ-менеджмента
- Обеспечить сбалансированный, единонаправленный коллективный стимул на выполнение стратегических целей
- Автоматически рассчитывать вознаграждение на основе факта исполнения KPI
- Предоставить прозрачный workflow согласования результатов
- Позволить моделировать вознаграждение до его официального утверждения

## 2. Роли и права доступа

Система поддерживает три роли с разграниченным доступом. Роль хранится в поле `system_role` таблицы `users`.

| **Роль** | **system_role** | **Видит** | **Может делать** |
|---|---|---|---|
| Администратор | admin | Всё | Полное управление системой: участники, карты, библиотека KPI, запускающие цели, справочники, архив |
| Согласующий | approver | Свою карту + карты подчинённых | Вносить факт, согласовывать / возвращать / отменять согласование строк карты |
| Участник | participant | Только свою карту | Вносить факт по своим KPI, моделировать запускающую цель |

> *Иерархия согласования: каждый участник имеет назначенного согласующего (поле `approver_id` в таблице `users`). Согласующий видит карты всех своих подчинённых.*

### 2.1 Навигация по ролям

Набор страниц меню зависит от роли. Навигация реализована через Next.js App Router (URL-based, deep links). Каждая страница имеет собственный URL.

| **Роль** | **Доступные разделы** |
|---|---|
| Admin | Дашборд · Лента событий · Участники · Библиотека KPI · Запускающие цели · Карты KPI · Справочники · Архив |
| Approver | Дашборд · Лента событий · Мои карты · На согласовании · Архив |
| Participant | Главная · Мои карты · Архив |

Все роли имеют доступ к странице «Мой профиль» через пункт меню в нижней части sidebar рядом с кнопкой «Выйти».

### 2.2 Аутентификация

Вход в систему осуществляется по рабочему email и паролю через Supabase Auth (`signInWithPassword`).

- Страница входа (`app/(auth)/login/page.tsx`): поле «Рабочий email» (type=email, placeholder `name@company.com`) + «Пароль»
- `useAuth.signIn(email, password)` передаёт email напрямую в Supabase Auth без преобразований
- При создании участника admin указывает рабочий email — он используется как идентификатор в Supabase Auth
- Для создания пользователей используется серверный Supabase-клиент с `service_role` и `persistSession: false`, чтобы не сбрасывать сессию админа
- Профиль загружается из таблицы `users` по `auth_id` с JOIN на `level_value_id` и `approver_id`
- Контекст `useAuth` предоставляет: `session`, `profile`, `loading`, `signIn`, `signOut`, `refreshProfile`, `isAdmin`, `isApprover`, `isParticipant`

### 2.3 Смена пароля (Мой профиль)

Каждый пользователь может сменить пароль через страницу «Мой профиль» (`app/(dashboard)/profile/page.tsx`):
- Форма: текущий пароль → новый пароль (мин. 6 символов) → повтор нового пароля
- Текущий пароль проверяется через повторный `signInWithPassword`
- Новый пароль устанавливается через `supabase.auth.updateUser({ password })`

### 2.4 Сброс пароля (Admin)

Администратор может сбросить пароль участника из модальной формы редактирования:
- Кнопка «Сбросить пароль» → диалог подтверждения → `supabase.auth.resetPasswordForEmail`
- Если отправка email не удалась — fallback: генерация временного пароля и отображение админу

## 3. Методы оценки исполнения KPI

Система поддерживает четыре метода оценки. Метод задаётся в библиотеке KPI и наследуется при добавлении в карту.

### 3.1 Метод «По шкале» (scale)

Диапазонная оценка. Фактическое значение (в % от цели: `fact / target * 100`) попадает в один из диапазонов.

Реализация (`calcScaleExecution` в `src/lib/calculations.ts`):
- Диапазоны сортируются по `sort_order`
- Для каждого диапазона проверяется: `[from ; to)` для всех кроме последнего, `[from ; to]` для последнего
- `range_to = null` или пусто → трактуется как бесконечность
- Тип `fixed` → возвращается `fixed_pct`
- Тип `proportional` → возвращается `fact / target * 100`, ограниченное верхней границей диапазона
- Если факт не попадает ни в один диапазон → 0%

### 3.2 Метод «Бинарный» (binary)

Два значения:
- Да (fact = 1) → 100%
- Нет (fact = 0) → 0%

В UI отображается как две кнопки «Да» / «Нет».

### 3.3 Метод «Дискретный» (discrete)

Таблица соответствия: целое значение факта → процент исполнения.

Реализация (`calcDiscreteExecution`):
- Факт приводится к целому числу (`parseInt`)
- Ищется точное совпадение в таблице `points`
- Если значение не найдено → возвращается `'not_found'`

### 3.4 Метод «Ручной» (manual)

Согласующий или admin вводит итоговый % исполнения вручную. Введённое значение сохраняется и как `fact_value`, и как `execution_pct`.

## 4. Составность KPI

KPI может быть простым или составным. Составные KPI агрегируют дочерние строки L2.

### 4.1 Составность 1 — Средневзвешенная (weighted)

KPI-L1 не имеет собственного целевого значения. Его % исполнения:

**% KPI-L1 = Σ (вес_j / 100 × % KPI-L2_j)**

Реализация (`getCompositeL1Execution` в `src/lib/calculations.ts`):
- Итерирует по дочерним L2-строкам
- Для каждой рассчитывает `getLineExecution`
- Суммирует: `(weight/100) * execution`

### 4.2 Составность 2 — Суммарная (additive)

Цель L1 = Σ целей L2, Факт L1 = Σ фактов L2.

% исполнения рассчитывается по методу оценки самого L1 (применяется `calcScaleExecution` к агрегированным значениям).

### 4.3 Простой KPI

KPI без дочерних записей. % исполнения рассчитывается непосредственно по методу оценки.

## 5. Карта KPI

### 5.1 Структура карты

Карта KPI — персональный документ участника за период. Содержит:
- Заголовок: участник, период (тип + год + подпериод), статус, дата создания
- Блок запускающей цели с переключателем Официальное / Моделирование
- Блок расчёта вознаграждения (формула с live-пересчётом)
- Индикатор суммы весов (с предупреждением если ≠ 100%)
- Предупреждение о незаполненных фактах
- Таблицу KPI-строк L1 с раскрываемыми строками L2
- Журнал изменений (Audit Log) — раскрываемый блок внизу

### 5.2 Параметры KPI-строки

| **Поле** | **Описание** | **Кто заполняет** |
|---|---|---|
| KPI (из библиотеки) | Название показателя | Admin при создании карты |
| Вес (%) | Удельный вес KPI в итоговой оценке | Admin при создании карты |
| Метод оценки | scale / binary / discrete / manual | Наследуется из библиотеки |
| Единица измерения | Из KPI библиотеки | Наследуется из библиотеки |
| Целевое значение | Плановый показатель | Admin (из библиотеки, можно переопределить) |
| Фактическое значение | Реальный результат | Участник или согласующий |
| % исполнения | Автоматический расчёт по методу | Система |
| Комментарий участника | Пояснение к факту | Участник |
| Комментарий согласующего | Пояснение при возврате | Согласующий |
| is_approved | Строка согласована | Согласующий |
| is_composite | Составной KPI | Admin |
| composite_type | weighted / additive | Admin |

### 5.3 Итоговый расчёт по карте

Реализация (`calcCardTotal` в `src/lib/calculations.ts`):

**% карты = Σ (вес_i / 100 × % KPI_i)**

- Строки с `execution_pct = null` или `'not_found'` пропускаются, но помечаются как `hasUnfilled`
- Возвращается объект: `{ result, isComplete, totalWeight }`

## 6. Запускающая цель

### 6.1 Назначение

Запускающая цель — пороговое условие, определяющее процент выплаты вознаграждения независимо от % исполнения индивидуальных KPI.

### 6.2 Мастер-карта запускающей цели

- Создаётся только Admin в разделе «Запускающие цели» (`app/(dashboard)/trigger-goals/page.tsx`)
- Содержит: название, описание, applicable_levels (CEO-1, CEO-2), период, набор KPI-строк из библиотеки
- Admin вводит официальный факт по каждой KPI-строке → автоматически рассчитывается % (факт / цель × 100)
- Официальный % исполнения (`official_execution_pct`) отображается на карточке цели
- Одна мастер-карта может быть привязана к нескольким индивидуальным картам (через `trigger_goal_id` в `kpi_cards`)

### 6.3 Отображение в индивидуальной карте

Блок запускающей цели (`TriggerGoalBlock`) в верхней части карты:

| **Режим** | **Описание** |
|---|---|
| Официальное | Показывает `official_execution_pct`. Используется в расчёте вознаграждения |
| Моделирование | Участник вводит свои факты по KPI запускающей цели. Пересчитывается предполагаемый %. Данные сохраняются в `user_trigger_goal_data`, но не влияют на официальный % |

### 6.4 Формула вознаграждения

**Вознаграждение = Оклад × Кол-во окладов × % запускающей цели / 100 × % инд. KPI / 100**

Реализация (`calcReward` в `src/lib/calculations.ts`):
```typescript
const components = [baseSalary, salaryMultiplier, triggerGoalPct / 100, individualKpiPct / 100]
const result = components.reduce((acc, val) => acc * val, 1)
```

Отображение в карте: блок `RewardBlock` с визуальной формулой (каждый компонент × оператор).

| **Параметр** | **Источник** | **Поле в БД** |
|---|---|---|
| Базовый оклад | Профиль участника | `users.base_salary` |
| Количество окладов | Профиль участника | `users.salary_multiplier` |
| % запускающей цели | Мастер-карта или моделирование | `trigger_goals.official_execution_pct` |
| % индивидуальных KPI | Итоговый % карты | `kpi_cards.total_execution_pct` |

## 7. Статус-машина карты KPI

### 7.1 Жизненный цикл

| **Статус** | **Код** | **Описание** | **Кто переводит** |
|---|---|---|---|
| Черновик | `draft` | Карта создана, данные ещё не зафиксированы | Система при создании |
| Активна | `active` | Участник вносит факты, авторасчёт работает | Admin при активации |
| На согласовании | `pending_approval` | Участник отправил карту. Ввод фактов зависит от `is_approved` строки | Participant (кнопка «На согласование») |
| Утверждена | `approved` | Все строки согласованы | Автоматически при согласовании всех строк L1 |
| Возвращена | `returned` | Одна или более строк возвращены | Approver (при возврате строки или карты) |

### 7.2 Диаграмма переходов

```
draft → active → pending_approval → approved
                                  ↘ returned → pending_approval (повторно)
```

### 7.3 Функции статусов в коде

- `cardStatusLabel(status)` — русское название статуса
- `cardStatusColor(status)` — цвет для badge

## 8. Workflow согласования

### 8.1 Пошаговый процесс

| **Шаг** | **Действие** | **Кто** | **Результат** |
|---|---|---|---|
| 1 | Вносит факт по KPI-строке (кнопка Edit2 → inline-форма) | Любой с доступом | Авторасчёт %, audit_log запись |
| 2 | Нажимает «На согласование» (видна если `isComplete` и статус draft/active/returned) | Participant или Admin | Статус → `pending_approval`, event создаётся |
| 3 | Нажимает ✓ (Check) по строке | Approver или Admin | `is_approved = true`, `approved_at`, `approved_by` |
| 4 | Нажимает ↩ (RotateCcw) по строке с обязательным комментарием | Approver или Admin | Статус карты → `returned`, event создаётся |
| 5 | Все L1-строки согласованы | Система | Статус карты → `approved` автоматически |

### 8.2 Отмена согласования (unapprove)

- Кнопка RotateCcw (оранжевая) появляется для approver/admin рядом с согласованной строкой
- При отмене: `is_approved = false`, очищаются `approver_comment`, `approved_at`, `approved_by`
- Если карта была в статусе `approved` — возвращается в `pending_approval`

### 8.3 Доступ к вводу факта

Поле `fact_source` удалено из UI. Доступ определяется по:
- `line.is_approved === true` → ввод заблокирован
- `card.status === 'approved'` → ввод заблокирован
- В остальных случаях ввод доступен всем с доступом к карте

### 8.4 Возврат карты целиком

Кнопка «Вернуть» в header карты (видна approver/admin при `pending_approval`) → модальное окно с обязательным комментарием → статус → `returned`.

## 9. Библиотека KPI

Централизованный справочник показателей (`app/(dashboard)/kpi-library/page.tsx`).

### 9.1 Параметры KPI в библиотеке

| **Поле** | **Описание** |
|---|---|
| Название KPI | Обязательное |
| Критерии исполнения / Описание | Текстовое описание (textarea) |
| Единица измерения | Фиксированный набор: руб., тыс. р., млн. р., млрд. р., %, шт., да / нет |
| Метод оценки | scale / binary / discrete / manual |
| Целевое значение | Числовое. Для discrete — целое положительное (DiscreteTargetField) |
| Период | Многоступенчатый: год → природа (за период / на дату) → пресет (Q1–Q4, H1–H2, год, произвольно) или конкретная дата |
| Диапазоны шкалы | Для метода scale: от%, до%, тип (fixed/proportional), % исполнения |
| Дискретные точки | Для метода discrete: факт (целое) → % исполнения |
| Свойства | Пары справочник:значение из динамических справочников |
| is_active | Мягкое удаление (скрытие из библиотеки) |

### 9.2 Связанные поля (auto-linkage)

- Выбор единицы «да / нет» → метод переключается на «бинарный»
- Выбор метода «бинарный» → единица устанавливается в «да / нет»
- Выбор метода «дискретный» → target=1, unit=шт.
- Смена метода с бинарного → единица сбрасывается в пустую

### 9.3 Период

Блок «Период» в форме создания/редактирования KPI:

1. **Шаг 1 — Отчётный год**: числовое поле
2. **Шаг 2 — Природа показателя**: toggle-сегмент «За период» / «На дату»
3. **Шаг 3**:
   - За период → сетка пресетов 4×2 (Q1–Q4, H1–H2, год, произвольно)
   - На дату → поле с датой (по умолчанию 31.12 отчётного года)
   - Произвольно → два поля «Начало» и «Конец»

При сохранении автоматически рассчитываются `period_date_from` и `period_date_to` из пресета.

### 9.4 Отображение периода в таблице

Функция `formatPeriodColumn`:
- `on_date` → дата в формате ДД.ММ.ГГГГ
- `for_period` + year → год
- `for_period` + Q/H → «Q1 2026» / «H2 2026»
- `for_period` + custom → «ДД.ММ.ГГГГ — ДД.ММ.ГГГГ»

### 9.5 Фильтры

Два ряда фильтров:
- **Ряд 1**: Поиск по названию, Метод оценки, Единица измерения, Период
- **Ряд 2**: Динамические фильтры по свойствам — для каждого справочника с `show_in_filters=true` отображается пара «Название: [select]»
- Кнопка «Сбросить фильтры» появляется при любом активном фильтре
- Счётчик: «Найдено: X из Y» или «Y показателей в библиотеке»

### 9.6 Копирование KPI

- Кнопка Copy в строке таблицы → открывает модальную форму с предзаполненными данными, заголовок «Копия KPI»
- Перед сохранением — проверка дубликатов по совокупности: название, метод, цель, единица, период (nature, year, preset), свойства
- При обнаружении дубликата — красный блок с описанием что изменить

### 9.7 Диапазоны шкалы в форме

- Каждый диапазон: От (%), До (%, пусто=∞), Тип (фиксированный/пропорционально), % исполнения
- Строка «В ед. изм.» под каждым диапазоном: пересчёт `[fromUnits — toUnits]` через `calcRangeBoundsInUnits`
- Для типа `proportional` поле % исполнения заблокировано (disabled), placeholder «авто»

### 9.8 Использование при добавлении в карту

- Admin выбирает KPI из библиотеки + указывает вес
- Метод, единица, целевое значение наследуются
- Диапазоны шкалы копируются в `card_line_scale_ranges`
- Дискретные точки копируются в `card_line_discrete_points`

- Кнопка «Импорт из Excel» → выбор файла .xlsx/.xls
- Парсинг через библиотеку `xlsx`: ищется столбец с заголовком «ФИО»
- Модальное окно импорта: таблица с ФИО, полями для email и пароля, выбор роли
- При подтверждении: серверный API route (service_role) + `INSERT` в users

### 10.4 Сброс пароля

В режиме редактирования кнопка «Сбросить пароль»:
- Диалог подтверждения с именем и email
- `supabase.auth.resetPasswordForEmail` → ссылка на email
- Fallback при ошибке: генерация временного пароля и отображение админу

## 12. Дополнительные механики

### 12.1 Журнал изменений (Audit Log)

- Каждое действие с картой фиксируется в таблице `audit_log`
- Типы: `fact_entered`, `submitted`, `line_approved`, `line_unapproved`, `card_approved`, `card_unapproved`, `returned`, `line_added`
- Поля: `entity_type`, `entity_id`, `action`, `new_value` (JSONB), `comment`, `performed_by`, `performed_at`
- В карте: раскрываемый блок «История изменений (N)» с хронологическим списком (последние 30 записей)
- Не редактируется и не удаляется

### 12.2 Лента событий

- Страница `app/(dashboard)/events/page.tsx` — для Admin и Approver
- Загружает последние 100 записей из таблицы `events` с JOIN на `created_by`
- Типы: `card_submitted`, `card_approved`, `card_returned`, `fact_entered`
- Иконки: Send (синий), CheckCircle (зелёный), RotateCcw (красный), User (teal)
- События создаются при отправке на согласование, утверждении и возврате

### 12.3 Архив

- `app/(dashboard)/archive/page.tsx` — read-only список карт со статусом `approved`
- Admin видит все, Approver — свои + подчинённых, Participant — только свои
- Колонки: Участник (для admin/approver), Период, % исполнения, Вознаграждение, Дата утверждения
- Массовая архивация — не реализована (запланировано)

### 12.4 Экспорт

- Зависимость `xlsx` подключена
- Используется только для импорта участников из Excel
- Экспорт карты в Excel / PDF — не реализован (запланировано)

## 15. Нефункциональные требования

### 15.1 Стек технологий

| **Компонент** | **Технология** |
|---|---|
| Frontend | Next.js 15 (App Router) + TypeScript strict |
| UI-компоненты | shadcn/ui (dark only) |
| Стили | Tailwind CSS v4 |
| Backend / БД | Supabase (PostgreSQL + Auth + RLS enforced) |
| Data fetching | TanStack Query v5 |
| Формы | React Hook Form + Zod |
| Хостинг | Vercel |
| Аутентификация | Supabase Auth (email/password) |
| Иконки | lucide-react |
| Excel | xlsx (только импорт участников) |
| Тестирование | Vitest + Testing Library |

### 15.2 Архитектурные решения

- Навигация через Next.js App Router (URL-based, deep links); каждая страница имеет собственный URL
- Алиас `@` → `src/` в `tsconfig.json`
- Два Supabase-клиента: браузерный (`src/lib/supabase/browser.ts`, anon key) и серверный (`src/lib/supabase/server.ts`, service_role, `persistSession: false`)
- RLS enforced как primary gate — application code как secondary
- Feature-based модульная структура: `src/features/` организован по бизнес-доменам
- Все расчёты KPI в `src/lib/calculations.ts` — единый файл бизнес-логики
- ViewModel + mapper pattern: компоненты принимают ViewModel-типы из `src/types/`, не сырые строки из БД
